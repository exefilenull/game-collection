# 攻略メモHTML生成手順(SKILL.md)

## このドキュメントについて

このドキュメントは、`game_notes/{category}/{hardware}/{title}.html`に攻略メモHTMLを新規生成する際に、どのAIエージェント(特定のプロバイダのAPIやツールに限定されない)でも読んで従える手順を定義する。特定のAI製品固有のツール名やコマンドをこの手順の必須要素として書くことはしない。前提として必要なのは、テキストエディタ相当の手段でHTMLファイルを書き出せることのみである。

`game_notes/`配下には既に29件のHTML実例が存在する。このドキュメントは、それら実例から抽出した構成・改行・カラーコード・文体のルールを明文化したものであり、実例そのものが「生きたテンプレート」である。新しい攻略メモを作成する際は、まず既存の実例(特に自分が担当するハード・カテゴリに近いもの)を1〜2件開いて確認し、以下のルールと照合しながら作成すること。

本セクション(HTML構成・改行・カラーコード・文体)は生成物の"体裁"を定義するものであり、対象レコードの抽出方法・生成先ディレクトリへの配置ロジック・`walkthrough_note_path`の紐付け手順は別セクションで扱う。

---

## 1. HTML構成規則

攻略メモHTMLは、`<!DOCTYPE html>`から`</html>`まで以下の構成を必ず持つこと。

- `<head>`
  - `<meta charset="UTF-8">`
  - `<title>`: ゲームタイトル(後述のカラーコード規則に従った注記を含めてよい)
  - `<style>`: 後述の固定CSS(**`white-space: pre-line`を含めないこと**。詳細は「2. 改行規則」参照)
- `<body>`
  - `<h1>`: タイトル見出し(後述のカラーコード規則に従う)
  - `<nav><ul>...</ul></nav>`: 目次。本文の見出し構造と対応する入れ子の`<ul><li><a href="#見出しid">...</a></li></ul>`
  - `<details open><summary><h2 id="...">...</h2></summary>...</details>`: 折りたたみ式セクション。`<details>`は見出しレベルに応じて`<h2>`→`<h3>`→`<h4>`と入れ子にできる(最大で`<h2>`→`<h3>`→`<h4>`の3階層まで使われた実例がある)

### テンプレート例

以下は既存実例(`game_notes/SIE/PlayStation1/バイオハザード3 LAST ESCAPE.html`等)から抽出した最小構成のテンプレートである。

```html
<!DOCTYPE html>
<html lang="ja">
<head>
<meta charset="UTF-8">
<title>ゲームタイトル</title>
<style>
body { font-family: Arial, sans-serif; background-color: #fffacd; padding: 20px; max-width: 900px; margin: 0 auto; }
nav { background: #fffacd; border: 1px solid #ccc; padding: 10px; margin-bottom: 20px; }
nav a { text-decoration: none; color: #333; }
details { margin: 10px 0; }
summary { cursor: pointer; }
</style>
</head>
<body>
<h1>ゲームタイトル（#）</h1>
<nav>
<ul>
<li><a href="#大見出しA">大見出しA</a>
<ul>
<li><a href="#中見出しA-1">中見出しA-1</a></li>
</ul>
</li>
<li><a href="#大見出しB">大見出しB</a></li>
</ul>
</nav>
<details open>
<summary><h2 id="大見出しA" style="display:inline">大見出しA</h2></summary>
<details open>
<summary><h3 id="中見出しA-1" style="display:inline">中見出しA-1</h3></summary>
アイテム1<br>
アイテム2<br><br>
</details>
<br><br><br>
</details>
<details open>
<summary><h2 id="大見出しB" style="display:inline">大見出しB</h2></summary>
アイテム3<br>
アイテム4<br><br><br>
</details>
</body>
</html>
```

`<style>`は上記の5行(`body`/`nav`/`nav a`/`details`/`summary`)を既存29件共通のCSSとしてそのまま使うこと。`<h1>`・`<nav>`は`<style>`にクラス指定が無いため、色付けは常にインラインの`<span style="color:#XXXXXX">`で行う(3章参照)。

`<details>`には常に`open`属性を付ける(既定で展開表示)。`<summary>`の中の見出し要素には`style="display:inline"`を付け、`<summary>`が見出し行として自然に見えるようにする。`id`属性は目次の`<a href="#...">`とアンカーで一致させる。見出しテキストに`/`や記号を含む場合、既存実例では`id`をそのまま日本語テキストや記号込みの文字列にしている(例: `id="スーパー-コンボ★"`のようにスペースをハイフンに変える程度の軽微な調整のみ)。

目次(`<nav>`)は、`<h1>`の直後に`<ul><li>`の入れ子リストとして置く。一部の古い実例では`<nav>`の前に`<h2>目次</h2>`という見出しを置いているが、直近の実例の大多数(29件中26件)は見出し無しで`<nav>`をそのまま置く構成であるため、迷う場合はこちらを既定とする。

## 2. 改行規則(重要)

**改行は明示的な`<br>`タグでのみ表現すること。`<style>`に`white-space: pre-line;`を含めてはならない。**

これは2026-07-23に確立されたルールである。かつて`white-space: pre-line`(ソース上の改行文字をそのまま可視化する方式)を採用していたが、`</summary>`と本文の間・`</h1>`と`<nav>`の間など、タグ同士の間にあるソース整形上の改行文字1つだけでも、ブラウザ上で余分な空行として表示されてしまう欠陥が判明した。この欠陥は`pre-line`下では「折りたたみ可能な空白」という通常のCSSの挙動が失われ、本来無視されるはずのブロック要素間の空白テキストノードが無視されなくなることに起因する。既存29件は全てこの欠陥を修正済みであり、新規生成物でも同じ欠陥を再発させないこと。

この方式では、`<style>`にwhite-space関連の指定を含めない(既定値`normal`のまま)。そのため、ソースコード上でタグをどう改行・インデントしても表示結果には影響しない。逆に言えば、**改行して見せたい箇所には必ず`<br>`タグを明示的に置く**必要がある(何もしなければ改行されない)。

### 見出しレベルに応じた区切り改行数

各`<details>`セクションの本文の末尾・`</details>`の直前には、そのセクションの見出しレベルに応じた本数の`<br>`を置き、視覚的な区切りを表現する。

| 見出しレベル | `</details>`直前に置く`<br>`の数 |
|---|---|
| 見出し大(`<h2>`セクション) | `<br>`×3 |
| 見出し中(`<h3>`セクション) | `<br>`×2 |
| 見出し小以下(`<h4>`セクション、およびそれより深い階層) | `<br>`×1 |

具体例(`game_notes/SIE/PlayStation2/塊魂.html`より):

```html
<details open>
<summary><h4 id="居間" style="display:inline">居間</h4></summary>
ダイザ<br>
...
ダイヤ　[珍品]<br>
</details>                    <!-- h4セクション: 直前は<br>×1(項目自体の<br>のみ) -->
```

```html
<details open>
<summary><h3 id="星をつくる" style="display:inline">星をつくる</h3></summary>
1<br>
...
月<br><br>
</details>                    <!-- h3セクション: 最後の項目の<br>に加えてもう1つ、合計<br>×2 -->
```

```html
<details open>
<summary><h2 id="ステージ" style="display:inline">ステージ</h2></summary>
<details open> ... </details>  <!-- 入れ子のh3セクション -->
<br><br><br>
</details>                    <!-- h2セクション: 子要素の後に<br>×3 -->
```

`<h2>`セクションが`<h3>`や`<h4>`を子に持つ場合は、子の`</details>`が終わった直後(親`<details>`の内容の末尾)に親の見出しレベル分の`<br>`を置く。`<h2>`セクションに本文が無い(未調査などで空の)場合も、`<br>`×3だけを置いて`</details>`で閉じてよい(例: `game_notes/SIE/PlayStation1/バイオハザード3 LAST ESCAPE.html`の「クリア特典」セクション)。

### 補足: 元の手書きメモの空行グループの保持

見出し区切りとは別に、元の手書きメモに存在した意図的な空行(同じ場面・同じグループで登場する複数アイテムのまとまりを示す区切りなど)は、リスト項目の途中に`<br><br>`(1行分の空白)として保持してよい。これはセクション末尾の区切りルールとは独立した、本文中の任意の空白表現であり、混同しないこと。

## 3. タイトルのカラーコード規則

`<h1>`のタイトルテキストには、末尾に`（#XXXXXX）`という注記を付ける(色指定が無い場合は`（#）`という空のプレースホルダーを付ける)。この注記はそのゲームの攻略メモ本文中で「注目アイテム」や「特筆事項」を示すためにインラインで使われている強調色(16進数カラーコード)を、タイトル自体に明記する既存の慣習である。

- **色指定がある場合**: `<h1>`のタイトルテキスト全体(末尾の`（#XXXXXX）`注記も含む)を`<span style="color:#XXXXXX">...</span>`で囲むこと。**注記部分だけを着色するのは誤り**であり、タイトル本体と注記を分けて一部だけ色付けしてはならない。

  ```html
  <h1><span style="color:#28A828">デッドライジング3☆（#28A828）</span></h1>
  ```

- **色指定が無い場合**: 既定で黒(`#000000`)として扱う。黒は`<body>`のCSSが定めるテキストの既定色そのものなので、実務上は`<span>`で囲まず素のテキストのままでよい(`（#）`という空注記のみ付ける)。

  ```html
  <h1>塊魂（#）</h1>
  ```

同じ色を使う場合、本文中で強調したい語句(既存実例では`☆`や`★`を伴うことが多い)にも同じ`<span style="color:#XXXXXX">語句☆</span>`を個別に適用する。目次(`<nav>`)側のリンクテキストも、対応する見出しが着色されている場合は同じ色で着色する(`<a href="#...">`の中身を`<span>`で囲む)。

## 4. 文体模倣の指針

既存の攻略メモは、プレイ状況・レビューサイトのような硬い文体ではなく、個人の手書きメモをそのままHTML化したような、率直でくだけたトーンで書かれている。新規生成時も以下の方針でこの雰囲気をできる範囲で模倣すること。

- 「ですます調」の説明文よりも、単語・体言止め・箇条書き中心の記述を優先する(既存29件は大半が単語の羅列)
- 個人的な感想・攻略メモ然としたコメントが本文に混じっていても、それを削って"正しい"文章に整形しない(例: `game_notes/SIE/PlayStation1/バイオハザード3 LAST ESCAPE.html`の「⇒HEAVY MODEのSランククリアで全5種類解放される」のような、矢印や注記を伴う一言コメント)
- `★`・`☆`・`☑`のような個人的な記号表記(レア度・入手済みチェック・お気に入り等を示す私的な記法)は、調査元の情報にそれらの記法が現れた場合はそのまま使い、汎用的な言い回しに置き換えない
- 内容の正確性そのものは本手順の保証範囲外である(調査結果の裏取りは行わない)。文体を似せることを優先し、事実確認に過度な労力をかける必要はない

## 5. プロバイダ非依存の前提

本ドキュメントはClaude Codeに限らず、どのAIエージェント・どのAIプロバイダのツールでも読んで従える手順として書かれている。手順の実行に特定のAI製品固有のツール名・API・拡張機能を前提とする記述は含めない。必要なのは、ハード名とゲームタイトルを手がかりに攻略情報を調査できる能力と、上記の構成規則に従ってテキストファイル(HTML)を書き出せる能力のみである。
